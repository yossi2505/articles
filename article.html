SQL injection flaw undermines read-only protections in deprecated Postgres MCP server

A recently published report by Datadog Security Labs details a SQL injection flaw in Anthropic’s reference Postgres MCP server, a component designed to give AI agents read-only access to PostgreSQL databases. The issue affects the archived server-postgres implementation, which has been deprecated but remains widely downloaded through package registries and container repositories.

The problem stems from how the server handles user-supplied SQL. Instead of enforcing a single safe statement, it passes raw input to the PostgreSQL client in a way that allows stacked queries separated by semicolons. That means an attacker or untrusted prompt input can inject a COMMIT statement to exit the read-only transaction and then execute arbitrary SQL outside the intended guardrail. In practice, this can turn a supposedly read-only database tool into a path for destructive writes.

The risk is especially relevant for teams experimenting with MCP servers in production-adjacent workflows, internal copilots, or agentic data access patterns. A connected agent could be manipulated into sending crafted SQL that drops schemas, alters data, or changes database session settings. The report also notes that session-level abuse is possible because pooled connections can retain modified state between requests, creating a denial-of-service condition for later queries even when database privileges are restricted.

Least-privilege database credentials can reduce the blast radius, but they do not fully solve the design weakness. Restricting the Postgres user may block unauthorized writes, yet it does not eliminate the possibility of session manipulation on reused connections. That makes the flaw more than a simple misconfiguration issue; it is a failure in how the server enforces isolation and safety for arbitrary SQL execution.

The documented fix is to prevent multiple statements by using prepared statements and to destroy the database session after each call so state cannot persist across requests. A patched version is available in Zed Industries’ maintained fork, while the deprecated reference package remained unpatched in the published NPM and Docker artifacts cited in the report. For defenders, the mitigation direction is clear: stop using the deprecated Postgres MCP server where write protection matters, move to a maintained patched implementation if needed, and ensure any database credentials exposed to AI tooling are tightly scoped.
